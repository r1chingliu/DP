# 技术方案

## 目标

项目需要满足三个要求：

1. 最终可在 iPhone 上使用
2. 支持持续迭代，便于通过 Git 在多台电脑接力开发
3. 先把核心体验做出来，再逐步接入 OCR、AI 和行情服务

## 技术栈

### 客户端

- `Expo`
- `React Native`
- `TypeScript`

### 本地服务端

当前已经落地为：

- `Node.js`
- `Express`
- `Multer`

这样做的原因很简单：这台机器的 `npm` 链路稳定，能够更快把真实截图上传和模型调用接起来。

### 后续持久化服务

后面如果要落数据库和历史快照，推荐：

- `PostgreSQL`
- `Prisma` 或其他 ORM
- `Redis` 作为缓存

## 环境变量

客户端：

- `EXPO_PUBLIC_API_BASE_URL`

服务端：

- `DASHSCOPE_BASE_URL`
- `DASHSCOPE_API_KEY`
- `DASHSCOPE_TEXT_MODEL`

不要把大模型密钥直接放进 React Native 客户端代码或提交到 Git。

## OCR 层

当前路线：

1. 前端选择券商截图
2. 上传到本地 `Express` 服务
3. 服务用 `tesseract.js` 做中文 OCR
4. 再把 OCR 原文交给 DashScope 文本模型结构化
5. 前端进入人工确认页
6. 确认后重建阵容

当前实现重点：

- 优先识别中文持仓名
- 提取市值、盈亏比例、成本价、市价
- 保留逐行置信度
- 无法确认的字段留空，不强行编造

## AI 层

AI 只做：

- 阵型推荐
- 位置推荐
- 角色标签
- 阵容点评

AI 不做：

- 投资建议
- 买卖判断
- 收益预测

实现原则：

- 规则优先
- 模型润色
- 输出可落库，不在页面实时漂移

## 当前前端架构

### `src/types`

定义资产、球场位置、阵容状态、OCR 识别结果等核心模型。

### `src/data`

保留模拟持仓数据，用于产品展示和无接口时的开发。

### `src/lib`

放排阵规则、持仓转换、手动换位、锁位等纯逻辑。

### `src/services`

放前端调用的接口层，包括 OCR 后端调用。

### `src/components`

放首页卡片、球场、替补席、球星卡、OCR 确认页等 UI 组件。
